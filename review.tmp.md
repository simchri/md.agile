## 1. Single-Source-of-Truth Violations

### 1.1 — `doc/checks.md` misstates which formatting checks have LSP quickfixes — Major — VERIFIED

**Where:** `doc/checks.md`, line "E001, E002, E003, E005, and E006 all have LSP quickfixes."

**Problem:** Cross-checked directly against the actual `REGISTRY` in
`crates/cli/src/lsp/quickfix/mod.rs`, which is explicitly commented as the
"Single source of truth for which `ErrorCode`s have a quickfix":

```rust
const REGISTRY: &[(ErrorCode, Builder)] = &[
    (ErrorCode::WrongIndentation, wrong_indentation::build),       // E002
    (ErrorCode::WrongBodyIndentation, wrong_body_indent::build),   // E003
    (ErrorCode::MissingSpaceAfterBox, missing_space_after_box::build), // E005
    (ErrorCode::BoxStyleInvalid, invalid_box::build),              // E006
    (ErrorCode::UppercaseX, uppercase_x::build),                   // E007
    (ErrorCode::UndefinedProperty, undefined_property::build),     // E008
    (ErrorCode::UndefinedAssignment, undefined_assignment::build), // E009
    (ErrorCode::MissingRequiredSubtasks, missing_subtasks::build), // E010
    // E001 OrphanedSubtask, E004 IncompleteParent: no quickfix
];
```

The doc claims **E001** has a quickfix (it explicitly does not, per the
code's own comment) and omits **E007** (which does). This is a hand-written
table drifting from the very code comment that calls itself the "single
source of truth" — ironic, given the doc restructure's stated goal was
reducing duplication/drift risk.

**Fix:** Correct the sentence to "E002, E003, E005, E006, and E007 all have
LSP quickfixes"; also note E008/E009/E010 have quickfixes too (currently
only stated individually, inconsistently, elsewhere in the same file).
Longer-term: consider a small test or doc-generation step asserting the
markdown table matches `REGISTRY`, since this has already drifted once,
same-day.

### 1.2 — `doc/config.md` incorrectly claims the LSP rejects invalid config — Major — VERIFIED

**Where:** `doc/config.md` "## Validation" section vs. `crates/cli/src/lsp/mod.rs:54-55`.

**Problem:** The doc states *"`agile check` (and the language server) reject
the config outright... reported on stderr and exit the process with status
1."* This is true only for the CLI (`cli/mod.rs` calls
`std::process::exit(1)` on a `Config::load` error). The LSP path does:

```rust
Config::load(config_path.parent().unwrap_or(config_path.as_path()))
    .unwrap_or_default()
```

Any config error — parse error, unknown key, bad group member, mismatched
`subtasks_allow_cancel` — is silently discarded and replaced with an
**empty** `Config::default()`. In the editor this means every
config-dependent check (E008–E013) silently goes quiet instead of surfacing
the actual config problem, which directly contradicts the new
documentation and is arguably a worse user experience than the CLI's loud
failure (a typo in `mdagile.toml` while editing would look like "all my
assignment/property markers stopped being validated," with no diagnostic
explaining why).

**Fix:** Either surface a diagnostic/error notification to the LSP client
when `Config::load` fails, or correct the doc to state the actual (silent
fallback) behavior and file a follow-up task to fix it, since silent
misbehavior in an editor is a worse failure mode than a CLI hard-exit.

## 2. Distributed / Implicit Assumptions

### 2.1 — `--base` ref-validation happens in the checker, not the git layer, based on convention rather than a type — Minor — VERIFIED

**Where:** `crates/cli/src/checker/mod.rs::check_authorization` vs.
`crates/cli/src/git/mod.rs::{ref_exists, file_content_at_ref}`.

**Problem:** `git::file_content_at_ref(dir, git_ref, path)` does **not**
validate that `git_ref` exists — it just returns `None` on any failure,
identical for "bad ref" and "ref valid but file not present" (a legitimate
untracked-file case). The distinction between these two cases is enforced
only in `checker::check_authorization`, which calls `git::ref_exists`
*first*, but **only when `base_ref` was explicitly passed by the caller** —
the default `"HEAD"` path skips this validation entirely (deliberately, so
a fresh repo with zero commits doesn't error out). This is a real design
constraint (documented in a comment on `InvalidRef` and in
`doc/assignment-validation.md`), but it's enforced purely by *conventions
at each call site* remembering to call `ref_exists` — nothing in the type
system stops a future call site from invoking `file_content_at_ref`
directly with a user-supplied ref and forgetting the validation, silently
reintroducing "typo'd ref treated as untracked file" instead of a hard
error.

**Fix:** Not urgent given there's currently only one call site
(`check_authorization`), but if a second `--base`-like flag is ever added
elsewhere, consider folding validation into a single wrapper function (e.g.
`file_content_at_validated_ref`) that always validates, rather than relying
on callers to remember `ref_exists` first.

## 3. Code Duplication

### 3.1 — `unrequired_quoted_subtask` duplicates `missing_required_subtasks`'s required-title collection and tree walk — Minor — VERIFIED

**Where:** `crates/cli/src/rules/unrequired_quoted_subtask/mod.rs`
(`collect_required`, recursive node walk) vs.
`crates/cli/src/rules/missing_required_subtasks/mod.rs` (the equivalent
required-title collection loop and its own recursive walk).

**Problem:** Both rules independently implement (a) the same recursive
"task → children → recurse" tree walk, and (b) the same "for each
`Marker::Property` on this node, look up `config.properties`, append each
`subtasks` entry if not already present" logic. The two copies currently
agree, but nothing enforces that. If "what counts as required" changes
(whitespace trimming, case sensitivity, inheriting from a parent property),
it's easy to update one copy and not the other — silently making E010 and
E011 disagree about which quoted subtasks are legitimate for the same task.

**Fix:** Extract a shared `fn required_subtasks(markers: &[Marker], config:
&Config) -> Vec<String>` used by both rules, and ideally a shared tree-walk
helper.

### 3.2 — `init_repo`/`commit_all` test helpers duplicated across unit and acceptance crates — Nitpick — VERIFIED (expected/low-risk)

**Where:** `crates/cli/src/git/tests.rs` vs.
`crates/cli/tests/acceptance/checks/e013.rs`.

**Problem:** Near-identical git-bootstrapping helpers exist in both places.
This is largely an unavoidable consequence of Rust's unit-test
(`#[cfg(test)]`, inside the crate) vs. integration-test (separate `tests/`
binary) boundary — there's no low-friction sharing mechanism without a
`dev-dependencies`-only test-support crate. Flagged only because if the two
copies drift (e.g. one adds `-b main` to `git init`, the other doesn't),
the two suites could behave differently across git versions/configs.

**Fix:** Not worth restructuring for this alone.

## 4. Fragile Interactions

### 4.1 — E013 matches old/new tasks by title alone, which collides on duplicate titles — **Critical** — VERIFIED BY DIRECT CODE READ

**Where:** `crates/cli/src/rules/unauthorized_completion/mod.rs`:

```rust
let old_status_by_title: HashMap<&str, &Status> = old
    .map(|items| flatten(items).into_iter().map(|n| (n.title, n.status)).collect::<Vec<_>>())
    .unwrap_or_default()
    .into_iter()
    .collect();
```

**Problem:** `flatten()` walks the *entire file* — every task and subtask,
at every nesting depth — into one flat list keyed only by title text, then
collects into a `HashMap` where a later duplicate **silently overwrites an
earlier one**. This is not a remote edge case: required subtasks
(`[Properties.X].subtasks`) are, by design, **the same literal title reused
across every task carrying that property** — this repo's own
`mdagile.toml` has `[Properties.foo] subtasks = ["bar", "baz"]`, so any file
with two-plus tasks tagged `#foo` will have multiple subtasks literally
titled `"bar"`.

Concrete failure scenario: task A's `"bar"` subtask was already `[x]` at the
base ref, task B's `"bar"` subtask was `[ ]` at the base ref. Both are `[x]`
in the working copy now. Because `HashMap::collect()` keeps whichever
`(title, status)` pair appears last during iteration, task A's `"bar"` may
be looked up against task B's *old* status (or vice versa) — this can
produce either:
- a **false negative**: an actually-new, unauthorized completion goes
  unflagged because it's matched against a same-titled sibling that was
  already done, or
- a **false positive**: an already-legitimately-completed subtask gets
  flagged as a fresh unauthorized completion.

Either way, this undermines E013's core correctness guarantee, in exactly
the scenario (`#property`-driven required subtasks) that this project's own
"missing required subtasks" feature (E010) actively encourages users to
create.

**No existing test exercises this.** Searched
`crates/cli/src/rules/unauthorized_completion/tests.rs`,
`crates/cli/src/checker/tests.rs`, and
`crates/cli/tests/acceptance/checks/e013.rs` for duplicate-titled
tasks/subtasks in a single file — zero hits.

**Fix:** Match nodes by a stable identity instead of bare title — e.g. a
`(parent task title chain, subtask title)` path, or restrict matching to
within the corresponding parent task (itself matched by title) rather than
flattening the whole file into one global namespace. Add a regression test
with two same-titled subtasks under different parent tasks, each with a
different old/new transition, asserting the check attributes the correct
old status to the correct node.

### 4.2 — LSP `validate()` shells out to git synchronously on every keystroke-triggered `didChange`, with no debounce — Minor — VERIFIED

**Where:** `crates/cli/src/lsp/mod.rs::Backend::validate`, invoked from
`did_open`/`did_change` (confirmed no debounce logic around
`did_change`), calling `checker::check_authorization_for_document` →
`git::{is_git_repo, current_identity, file_content_at_ref}`, all of which
use blocking `std::process::Command::output()`/`status()`.

**Problem:** Each `validate()` call now spawns ~3-4 blocking `git`
subprocesses (repo check, two `git config` reads, one `git show`)
synchronously inside an `async fn`. The multi-threaded tokio runtime avoids
a hard deadlock, but on typing-heavy sessions across multiple open
`.agile.md` files, this adds real per-keystroke latency and thread-pool
pressure that the previous, purely in-memory `checker::run` path didn't
have.

**Fix:** Wrap the git-shelling calls in `tokio::task::spawn_blocking`, or
add debounce logic to `did_change`.

same-titled tasks/subtasks (realistic via a shared `#property`'s required
subtasks) with independently-different old→new status transitions,
asserting each is checked against its own correct prior status.

any) the LSP produces when the discovered config fails to parse or fails
group/property validation. Currently nothing would catch a regression (or
confirm the current silent-empty-config fallback) either way.
